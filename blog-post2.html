Rethinking @Input and @Output in Angular

I wasted weeks confused by component communication until one simple mindset shift changed everything.

R

Rupesh

Frontend Developer

I wasted weeks learning @Input and @Output in Angular. I was stuck in this rigid mindset: if you want to send data from parent to child → use @Input, and then child back to parent → use @Output.

I kept repeating that rule in my head, but it never really clicked how to use it in real projects. Things only changed when I started thinking in a completely different way.

The "Empty Component" Mindset

Instead of abstractly thinking "I need @Input/@Output," I started thinking in terms of empty components.

Imagine you have a button that you want to use in multiple components, in multiple places. The button design is the same everywhere, but the text (label) will change depending on where you use it. So what do we do?

The Strategy:

  1. Create a button component that is basically empty.
  2. Keep the structure and styling inside the button component.
  3. Send the specific content/label from the parent wherever we use it.

That’s where @Input actually makes sense. In the child (button) component, we create a property that will act as a placeholder to receive data from the parent.

button.component.ts (Child) 
@Input() dataFromParent = '';

Now dataFromParent is empty by default, and it lives in the child. Whenever we use this button, we bind any data from the parent to it:

app.component.html (Parent) 
<!-- Parent passing variable 'data' to child -->
<app-child [dataFromParent]="data"></app-child>

Here, data is just a variable in the parent component. Whatever value it holds will show up inside the button.

The Aha Moment: Once I started thinking like this—"make empty, reusable components and feed them data from the parent"—@Input finally started to feel natural.

I’m still not perfect with this stuff, but this mindset shift saved me a lot of frustration. If you're stuck memorizing the syntax, try focusing on the flow of data instead.
